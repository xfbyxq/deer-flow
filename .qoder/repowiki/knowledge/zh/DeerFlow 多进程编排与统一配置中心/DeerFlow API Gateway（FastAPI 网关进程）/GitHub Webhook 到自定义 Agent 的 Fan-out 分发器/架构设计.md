模块以 dispatcher.fanout_event 为唯一对外入口，被 app.gateway.routers.github_webhooks 调用；内部按职责拆分为小模块：
- registry.py：扫描 {base_dir}/users/{user_id}/agents/ 与遗留 {base_dir}/agents/，解析每个 agent 的 config.yaml 中 github.bindings[].triggers，构建 (repo, event) -> [GitHubAgentMatch] 索引，并用 (user_id, agent_name, config.yaml mtime) 三元组签名做进程内缓存，冷路径走 asyncio.to_thread 避免阻塞事件循环。
- triggers.py：纯函数实现 per-event 触发门控（action 白名单、allow_authors、边界感知的 @mention 匹配），通过 _resolved_trigger 将 binding 覆盖与 DEFAULT_TRIGGERS 字段级合并。
- prompts.py：按事件类型把 payload 渲染成描述性 prompt 字符串，不直接写回 GitHub，仅作为 user message 传入 agent。
- identity.py：用 uuid.uuid5 + 固定命名空间生成 (repo, number, agent_name) 确定的 langgraph thread id，以及从 payload 提取 (repo, number)。
- run_policy.py：在 import 时通过 register_policy() 向全局 CHANNEL_RUN_POLICY["github"] 注册非交互式、fire-and-forget、默认 recursion_limit=250 的策略，并在运行时注入安装令牌。
- app_auth.py：两阶段 GitHub App 认证——RS256 签名的 App JWT（10min TTL）+ 每 installation 的 1h access token，后者带 per-installation asyncio.Lock 的进程内缓存（5min 提前刷新）。
- __init__.py：副作用导入 run_policy，确保任何子包被 import 即完成策略注册。
依赖方向单向：dispatcher -> {registry, triggers, prompts, identity}，run_policy -> app_auth，registry -> triggers；无反向依赖。
